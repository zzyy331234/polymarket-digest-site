# MRAS v2 最终方案（面向自动化运营）

## 目标
做一套 **可长期自动化运行、尽量少人工盯盘** 的 Polymarket 策略系统。

不是“给几个信号”，而是完整闭环：
- 自动采集
- 自动特征计算
- 自动打分 / 过滤
- 自动生成候选
- 自动推送提醒
- 后续可接自动下单
- 能做复盘与持续调参

---

## 一、参考思路（结合现状 + 外部公开资料整理）

我参考了几类公开思路：
1. **Prediction market mean reversion**
   - 核心思想：极端赔率经常存在短期过冲，适合做回归
2. **Momentum / trend following**
   - 适合流动性好、事件热度上升、概率连续迁移的市场
3. **Orderbook / liquidity aware trading**
   - 盘口深度、spread、流动性、成交量变化决定信号是否可交易
4. **Prediction market 特有约束**
   - 临近事件、低赔率市场、长尾事件、跳价市场，需要单独处理

对你现在这套系统来说，正确方向不是“纯因子打分”，而是：

> **先判断市场属于哪种可交易结构，再用对应子策略处理。**

也就是继续保留 MRAS 思路，但要升级为真正的“策略系统”。

## 二、v2 总体架构

## 1. 数据层（Data Adapter）
新文件：`polymarket/data_adapter.py`

职责：
- 从 SQLite 读取市场和价格历史
- 统一生成标准 feature
- 对外只输出“干净特征”

### 输入
- `markets`
- `price_history`
- 后续可加 `orderflow`
- 后续可加 event 元数据

### 输出特征（建议最少集）
```python
{
  "token_id": str,
  "question": str,
  "yes_price": float,
  "no_price": float,
  "liquidity": float,
  "volume": float,
  "spread": float | None,
  "hours_to_event": float | None,

  "chg_1h": float | None,
  "chg_4h": float | None,
  "chg_24h": float | None,
  "chg_7d": float | None,

  "rsi_14": float | None,
  "volatility_24h": float | None,
  "trend_slope_4h": float | None,
  "trend_slope_24h": float | None,

  "pct_rank_7d": float | None,
  "pct_rank_30d": float | None,

  "micro_move_score": float | None,
  "tick_density": float | None,
}
```

### v2 原则
- 没有真实 24h / 7d 数据，就不要伪造
- 缺数据就降级，不要假装完整
- 特征层必须和策略层解耦

---

## 2. 策略层（Signal Engine）
新文件：`polymarket/mras_v2.py`

职责：
- 输入 feature
- 输出 signal
- 只做策略，不碰数据库、不碰 UI

### 统一输出格式
```python
{
  "token_id": str,
  "question": str,
  "regime": "trend" | "mean_revert" | "contrarian" | "carry_no" | "skip",
  "direction": "YES" | "NO" | None,
  "confidence": float,
  "position": "small" | "medium" | "large" | None,
  "quality": float,
  "risk_score": float,
  "reasons": list[str],
  "risk_flags": list[str],
  "features": {...},
}
```

## 三、v2 不再只用 4 个 regime，建议升级为 5 类

### 1. `trend`
适合：
- 流动性较好
- 24h / 7d 同方向
- 走势不是单跳价，而是连续迁移

### 2. `mean_revert`
适合：
- 赔率到极端区间
- 波动率抬升后回归
- 适合短线反弹 / 回落

### 3. `contrarian`
适合：
- 市场共识高度拥挤
- 动能开始衰减
- 盘口/价格显示反向信号

### 4. `carry_no`
这是我建议新增的。

适合：
- YES 概率极低
- 事件逻辑稳定
- 市场长期偏向 NO
- 不追波动，做“低赔率稳定 NO / 不做 YES”风格

这类市场在 prediction market 很常见，不能一股脑塞到 trend 或 mean_revert 里。

### 5. `skip`
直接不做：
- 流动性差
- spread 过大
- 临近事件
- 数据不完整
- 微小波动噪音

---

## 四、每个 regime 的推荐规则

## A. Trend
### 触发条件
- `liquidity >= 50k`
- `chg_24h` 与 `chg_7d` 同向
- `trend_slope_4h` 与 `trend_slope_24h` 同向
- `volatility_24h` 不过高
- `yes_price` 不在极端赔率区（默认排除 `<0.03` 和 `>0.97`）

### 方向
- `chg_24h > 0` → YES
- `chg_24h < 0` → NO

### 风险过滤
- 临近事件不追
- 单根跳价不算趋势
- 超低赔率市场不走 trend

---

## B. Mean Revert
### 触发条件
- `pct_rank_7d <= 0.1` 或 `>= 0.9`
- `rsi_14` 极端
- `volatility_24h` 高于中位数
- 价格偏离明显，但开始停止单边延续

### 方向
- 低分位 + 动能减弱 → YES 反弹
- 高分位 + 动能减弱 → NO 回落

### 风险过滤
- 不是所有极端价格都回归
- 必须要有“停止继续恶化”的迹象

---

## C. Contrarian
### 触发条件
- 高共识（如 yes_price >= 0.65）
- 但近期动能开始反向
- spread 没有明显恶化
- 盘口 / slope 支持反向

### 方向
- crowded YES fading → NO
- crowded NO fading → YES

### 风险过滤
- 只在高流动性市场做
- 必须比 trend 更严格

---

## D. Carry_NO（新增）
### 触发条件
- `yes_price <= 0.03` 或 `<= 0.05`
- 市场长期逻辑稳定
- 24h / 7d 波动很小
- 不是 event 临近的剧烈重估期

### 作用
这类市场不一定适合“搏反弹”，也不适合“趋势追涨”。
它更像：
- 稳定偏 NO 的被动优势市场
- 或者明确标记为“仅观测，不交易 YES”

### 这类 regime 的意义
把现在大量极低赔率市场从 trend 里剥离出去。

## 五、信号打分体系（v2）

v2 不建议继续用“hit 越多越直接顶满”。

### 建议拆成三层分数

#### 1. `setup_score`
市场结构是否适合该策略
- 流动性
- spread
- 数据完整性
- 时间窗口完整性
- 赔率区间适配

#### 2. `edge_score`
策略边际优势是否清晰
- 趋势强度 / 回归强度 / 共识拥挤程度
- 多窗口是否一致
- 极端程度是否足够

#### 3. `execution_score`
这个信号现在能不能做
- 事件时间
- spread
- 波动率
- 噪音水平
- 是否连续重复信号

### 最终分数
```python
final_conf = 0.4 * setup_score + 0.4 * edge_score + 0.2 * execution_score
```

### 仓位建议
- `>= 0.78` → large
- `>= 0.64` → medium
- `>= 0.52` → small
- `< 0.52` → skip

这样 large 会少很多，更符合实战。

## 六、自动化运营需要的“运营层”

如果目标是**自动化运营**，策略本身还不够，需要运营规则。

新文件建议：
- `polymarket/ops_rules.py`
- `polymarket/portfolio_manager.py`

### 运营层要做的事

#### 1. 去重与冷却
- 同一市场 6h 内不重复推送同方向信号
- 同一市场 regime 切换时才重新提醒

#### 2. 组合风险
- 单一题材上限（如全是世界杯）
- 单个事件簇上限
- 同类低赔率 NO 不要堆满

#### 3. 仓位控制
- 单笔固定上限
- 总风险预算上限
- 连续亏损后自动降档

#### 4. 执行门槛
- spread 太大不推
- 临近事件不推
- 数据窗口不完整不推

#### 5. 复盘闭环
- 每个信号存档
- 赛后自动打标签：赢/亏/未成交/放弃
- 每周生成 regime 胜率报告

  - v1 signals

  - 差异对比

- 先只用 v2 做提醒，不做自动执行
- 连续观察 3~7 天

- `polymarket_forecast.py` 改为调用 v2

## 十、最终判断

如果你的目标是：
> **最终直接能自动化运营，不需要你一直盯着**

那答案很明确：

**应该做 MRAS v2，而且要把它设计成“策略系统”，不是一段 calculate_signal 函数。**

重点不只是“预测方向”，而是：
- 先判断这个市场能不能做
- 属于哪类结构
- 有没有执行价值
- 会不会和已有风险重复
- 是否值得推送 / 下单

这才是自动化运营真正能用的形态。
